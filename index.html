单元测试

发表于 2022-03-19 更新于 2022-03-27

Android测试的分类:

单元测试( Junit4 , Junit5,Mockito,PowerMockito,Robolectric)

UI测试 (Espresso, UI Automator)

压力测试 (Monkey)

单元测试的目的: 方法测试

单元测试框架

Junit4 (官方)

是事实上的Java标准测试库，并且它是JUnit框架有史以来的最大改进，其主要目标便是利用Java5的Annotation特性简化测试用例的编写。

  • 缺点：无法单独对Android UI，一些类进行操作，与原生JAVA有一些差异。
  • https://pdai.tech/md/develop/ut/dev-ut-x-junit.html

集成

dependencies { … testImplementation ‘junit:junit:4.12’ }

常用注解:

@Test 指明这是一个测试方法 (@Test注解可以接受2个参数，一个是预期错误 expected，一个是超时时间timeout，

@Test(expected = IndexOutOfBoundsException.class),

@Test（timeout = 1000)

@Before 在所有测试方法之前执行

@After 在所有测试方法之后执行

@BeforeClass 在该类的所有测试方法和@Before方法之前执行 （修饰的方法必须是静态的）

@AfterClass 在该类的所有测试方法和@After 方法之后执行（修饰的方法必须是静态的）

@Ignore 忽略此单元测试

常用测试方法

assertEquals(expected, actual) 判断2个值是否相等

assertEquals(expected, actual,errormsg) 判断2个值是否相等

assertEquals(expected, actual,tolerance) tolerance 偏差值

assertEquals(expected, actual,tolerance,errormsg) tolerance 偏差值

自定义junit 规则

实现TestRule接口并重写apply方法

在想要的测试类中使用@Rule注解声明使用JsonChaoRule即可（注意被@Rule注解的变量必须是final的）

音视频基础

发表于 2022-03-19
前言: 目前音视频中使用最多的编解码方式 H264+AAC

相关名词:

  1. 帧 Frame

    1. I帧 : 相对完整帧( 去掉了人眼不敏感的色度,保留了人眼敏感的亮度)
    2. P帧
    3. B帧

  2. 编码 (编码最重要的目的是 为了进行数据压缩,以此来降低数据传输和存储的成本)

    1. 软编解码 : 利用软件本身或使用CPU对音视频进行编解码

      优点: 兼容性好

      缺点 : CPU占用率高, app占用率高

      软解码器开头通常是 OMX.google

    2. 硬编解码 : 使用非CPU进行编解码,如显卡GPU,专用DSP芯片,厂商芯片.

      优点: 编解码速度快且效率高,CPU占用率低

      缺点:兼容性不好

      硬解码器开头通常是OMX.[hardware_vendor]

    3. 获取手机所支持的编解码器

    4. MediaCodec 硬编解码

      1. MediaCodec调用的是系统注册过的编解码器，硬件厂商把自己的硬编解码器注册到系统中就是硬编解码，如果硬件厂商注册的是软编解码就是软解码。往往不同的硬件厂商是不一样的。然后MediaCodec负责调用。

        https://developer.android.google.cn/reference/android/media/MediaCodec?hl=en

音频基础

声波三要素: 频率, 振幅 , 波形

声音信号 -> 电信号 -> 数字信号(二进制)

为了将模拟信号数字化, 需要进行

采样:在时间轴上对信号数字化；

量化:在幅度轴上对信号数字化；

编码:按一定格式记录采样和量化后的数字数据。

声道:声道数也就是声音录制时的声音源数量或者回放时相应的扬声器数量

数据格式:

  1. 通常音频所说的裸数据就是 PCM

描述一段 PCM 数据一般需要以下几个概念：量化格式(sampleFormat)、采样率（sampleRate）、声道数 (channel) 。

  1. 编码算法(压缩):

PCM 、WAV、AAC 、MP3 、Ogg

  1. WAV 编码就是在 PCM 数据格式的前面加了 44 个字节，分别用来存储 PCM 的采样率、声道数、数据格式等信息。特点: 音质好，大量软件支持。
  2. MP3 具有不错的压缩比，使用 LAME 编码 （MP3 编码格式的一种实现）的中高码率的 MP3 文件，听感上非常接近源 WAV 文件
  3. AAC 是新一代的音频有损压缩技术，它通过一些附加的编码技术(比如 PS 、SBR) 等，衍生出了 LC-AAC 、HE-AAC 、HE-AAC v2 三种主要的编码格式。一般在音视频通话，直播中，短视频，以及大部分的视频都是用aac编码。
  4. Ogg 是一种非常有潜力的编码，在各种码率下都有比较优秀的表现，尤其是在中低码率场景下。 可以用比 MP3 更小的码率实现比 MP3 更好的音质

  1. 播放PCM的两种方式

    在Android中有两种方法：一种是使用java层的AudioTrack方法，一种是使用底层的OpenSLES直接在jni层调用系统的OpenSLES的c方法实现。

  2. 音频采集的两种方式

    MediaRecorder 和 AudioRecord . MediaRecorder最终还是调用AudioRecord.

    区别:MediaRecorder 录制后的音频数据是经过压缩的, 需要设置编码器 ,可以用播放器播放.

    AudioRecord 录制的是PCM格式的数据.,需要用AudioTrack播放.

AIDL

发表于 2022-03-17 更新于 2022-03-18

调用过程是同步的.

如何使 AIDL 的调用过程是异步的呢？

只需要把调用放到非 UI 线程即可，如果要对调用的返回做 UI 更新的话，再通过 Handler 处理即可。

默认支持的数据类型:

基本数据类型: byte，short，int，long，float，double，boolean，char

String 类型

CharSequence类型

List类型：List中的所有元素必须是AIDL支持的类型之一，或者是一个其他AIDL生成的接口，或者是定义的parcelable（下文关于这个会有详解）。List可以使用泛型。

Map类型：Map中的所有元素必须是AIDL支持的类型之一，或者是一个其他AIDL生成的接口，或者是定义的parcelable。Map是不支持泛型的。

定向tag:

Java 中的基本类型和 String ，CharSequence 的定向 tag 默认且只能是 in 。

in 为定向 tag 的话表现为服务端将会接收到一个那个对象的完整数据，但是客户端的那个对象不会因为服务端对传参的修改而发生变动；out 的话表现为服务端将会接收到那个对象的参数为空的对象，但是在服务端对接收到的空对象有任何修改之后客户端将会同步变动；inout 为定向 tag 的情况下，服务端将会接收到客户端传来对象的完整信息，并且客户端将会同步服务端对该对象的任何变动。
